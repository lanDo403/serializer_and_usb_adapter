Система высокоскоростного обмена данными на базе ПЛИС с использованием интерфейса USB 3.0

## Назначение документа

Этот файл является одновременно:
- кратким описанием проекта;
- рабочей спецификацией RTL-дизайна;
- базой для построения тестбенчей и критериев верификации.

При разработке и верификации необходимо ориентироваться в первую очередь на этот документ, а не на случайные особенности текущей реализации. Если реализация расходится со спецификацией, корректироваться должен RTL либо в документе должна быть явно зафиксирована новая версия требований.

## Назначение проекта

Проект предназначен для приема входного потока данных на ПЛИС `Xilinx Spartan-6` с последующей передачей данных на ПК через `FTDI FT601` в режиме `245 Synchronous FIFO`.

Целевой тракт данных:

`входная плата -> GPIO Spartan-6 -> упаковка 8 бит в 32 бита -> асинхронный FIFO -> FT601 -> ПК`

Дополнительно реализован обратный тракт:

`ПК -> FT601 -> внутренний RX FIFO -> внутренняя логика FPGA`

Обратный тракт используется не для широкого пользовательского потока данных, а для приема внутренних служебных слов и команд управления.

## Аппаратный контекст

- Плата FPGA: `Callisto S6`
- ПЛИС: `Xilinx Spartan-6`
- USB мост: `FTDI FT601`
- Среда разработки RTL: `Xilinx ISE 14.x`
- Базовый язык описания: `Verilog HDL`
- Основные справочные документы:
  - `docs/DS_FT600Q-FT601Q-IC-Datasheet.pdf`
  - `docs/AN_421_FIFO_Bus_Master_For FT60x.pdf`
  - `docs/spartan6_mst_fifo32_1.1/...` как reference design FTDI

## Границы верхнего модуля

Финальный верхний модуль проекта: `fpga/fifo_top.v`

На верхний уровень должны выходить только физические сигналы:
- входной интерфейс со стороны входной платы;
- сигналы FT601.

Внутренние признаки состояния FIFO, ошибки, служебные команды и служебные регистры не должны выводиться наружу отдельными портами верхнего модуля.

## Внешние интерфейсы

### Интерфейс входного потока в FPGA

Сигналы верхнего уровня:
- `LVDS_CLK`
- `LVDS_DATA[7:0]`
- `LVDS_STROB`
- `FPGA_RESET`

Важно:
- несмотря на историческое имя модуля `LVDS.v`, в текущем RTL на FPGA принимаются уже однополярные сигналы `LVCMOS33`;
- преобразование дифференциальных линий выполняется внешней аналоговой платой;
- RTL не выполняет декодирование дифференциальных пар внутри Spartan-6.

### Интерфейс FT601

Сигналы верхнего уровня:
- входы от FT601: `CLK`, `RESET_N`, `TXE_N`, `RXF_N`
- выходы в FT601: `OE_N`, `WR_N`, `RD_N`
- двунаправленные шины: `BE[3:0]`, `DATA[31:0]`

Активный уровень служебных сигналов FT601:
- `TXE_N = 0` — FT601 готов принимать данные от FPGA;
- `RXF_N = 0` — в FT601 есть данные для чтения FPGA;
- `WR_N = 0`, `RD_N = 0`, `OE_N = 0` — активные низким уровнем сигналы управления интерфейсом.

## Тактовые домены и сброс

В проекте есть два основных домена:

1. Домен записи входного потока
- такт: `lvds_clk`
- источник: `LVDS_CLK` через модуль `LVDS.v`
- основные модули домена:
  - `LVDS.v`
  - `packer8to32.v`
  - write-сторона `fifo_dualport.v`
  - write-сторона `sram_dp.v`

2. Домен FT601
- такт: `CLK`
- источник: тактовый сигнал от FT601
- основные модули домена:
  - read-сторона TX FIFO
  - `fifo_fsm.v`
  - RX FIFO
  - `fifo_rx_ctrl.v`

Правило по фронтам:
- write-domain логика входного тракта работает по своему входному такту и не переводится на `negedge` из-за требований FT601;
- FT601-domain логика в `fifo_fsm.v` использует `negedge CLK`, чтобы разделить по полутактам:
  - чтение TX FIFO / SRAM на `posedge CLK`;
  - захват слова из FIFO и выдачу управляющих сигналов FT601 на `negedge CLK`.

Сбросы:
- `FPGA_RESET` используется как асинхронный сброс для write-domain логики;
- `RESET_N` используется как сброс со стороны FT601;
- для внутреннего RX FIFO используется комбинированный сброс:
  - `rx_fifo_rst_n = RESET_N & ~FPGA_RESET`
  - `rx_fifo_rst_a = ~rx_fifo_rst_n`

## Архитектура проекта

### 1. Модуль `LVDS.v`

Назначение:
- буферизация входных сигналов `LVDS_CLK`, `LVDS_DATA`, `LVDS_STROB`;
- регистрация входных данных и строба во входном тактовом домене.

Требуемое поведение:
- `data_i` и `strob_i` захватываются по `posedge clk_buf`;
- на выход формируются:
  - `data_o` — зарегистрированные 8-битные данные;
  - `strob_o` — зарегистрированный признак валидности;
  - `clk_o` — буферизованный такт.

Замечание:
- модуль не анализирует содержимое входных байтов;
- валидность определяется только сигналом `strob_i/strob_o`.

### 2. Модуль `packer8to32.v`

Назначение:
- упаковка последовательности валидных 8-битных байтов в 32-битные слова.

Требуемое поведение:
- байт принимается только при `valid_i = 1`;
- каждые четыре валидных байта собираются в одно 32-битное слово;
- порядок байтов:
  - первый байт -> `data_o[7:0]`
  - второй байт -> `data_o[15:8]`
  - третий байт -> `data_o[23:16]`
  - четвертый байт -> `data_o[31:24]`
- `valid_o` активируется на один такт только в момент готовности полного 32-битного слова.

Ограничение текущей архитектуры:
- передача неполного последнего слова не поддерживается;
- интерфейс FT601 со стороны передачи использует только полные 32-битные слова, поэтому `BE` на передачу фиксируется как `4'hF`.

### 3. Модуль `fifo_dualport.v`

Назначение:
- асинхронный FIFO между write-domain (`lvds_clk`) и FT601-domain (`CLK`).

Требуемое поведение:
- FIFO использует бинарные и Gray-указатели;
- указатели синхронизируются между доменами через двухтактные синхронизаторы;
- формируются флаги:
  - `full`
  - `empty`
  - `overflow`
  - `underflow`.

Требования к флагам:
- `overflow` — sticky-флаг, фиксирует попытку записи в полный FIFO;
- `underflow` — sticky-флаг, фиксирует попытку чтения из пустого FIFO;
- для внутреннего RX FIFO детектор `underflow` может быть отключен параметром `USE_UNDERFLOW = 0`, если чтение из пустого FIFO исключено архитектурой.

### 4. Модуль `sram_dp.v`

Назначение:
- двухпортовая синхронная память, используемая как хранилище для FIFO.

Требуемое поведение:
- запись выполняется по `posedge wr_clk` при `wen = 1`;
- чтение выполняется по `posedge rd_clk` при `ren = 1`;
- данные чтения появляются на выходе синхронно, с задержкой относительно команды чтения.

Следствие для верификации:
- проверка read-path должна учитывать синхронную задержку чтения памяти.

### 5. TX-тракт: FPGA -> FT601

TX-тракт состоит из следующих блоков:
- `LVDS.v`
- `packer8to32.v`
- TX FIFO (`fifo_dualport.v` + `sram_dp.v`)
- `fifo_tx_ctrl.v`
- `fifo_fsm.v`

Назначение `fifo_tx_ctrl.v`:
- блокировать прием новых слов в TX FIFO после ошибок до явной очистки;
- синхронизировать служебные признаки из домена `CLK` в домен `lvds_clk`.

Требуемое поведение TX-тракта:
- запись из `packer8to32` в TX FIFO разрешена только при отсутствии sticky-ошибок;
- при попытке записи в полный TX FIFO должна фиксироваться ошибка TX;
- при попытке некорректного чтения TX FIFO должна фиксироваться ошибка TX;
- после фиксации sticky-ошибки прием новых слов из `packer8to32` блокируется до очистки ошибки через RX-команду.

### 6. RX-тракт: FT601 -> FPGA

RX-тракт состоит из следующих блоков:
- `fifo_fsm.v`
- RX FIFO (`fifo_dualport.v` + `sram_dp.v`)
- `fifo_rx_ctrl.v`

Назначение RX-тракта:
- прием 32-битных слов и `BE[3:0]` от FT601;
- временное буферирование принятых слов;
- передача слов во внутреннюю служебную логику FPGA.

Требуемое поведение:
- принятые от FT601 данные сохраняются во внутренний RX FIFO в формате `{BE, DATA}`;
- RX FIFO автоматически дренируется внутренним управляющим модулем `fifo_rx_ctrl.v`;
- пользовательские внешние порты для чтения RX FIFO на верхнем уровне не предусмотрены.

### 7. Модуль `fifo_fsm.v`

Назначение:
- управление интерфейсом FT601 в режиме `245 Synchronous FIFO`.

Основные функции:
- передача данных из TX FIFO в FT601;
- прием слов из FT601 в RX FIFO;
- управление `OE_N`, `WR_N`, `RD_N`;
- управление направлением шины `DATA/BE` через `drive_tx`.

Требуемое поведение передачи в FT601:
- передача допускается только при `TXE_N = 0` и `empty_fifo = 0`;
- FPGA должна выставлять `DATA` и `BE` только когда `drive_tx = 1`;
- при передаче полного 32-битного слова `BE = 4'hF`.
- передача выполняется burst-режимом:
  - после запроса первого слова из TX FIFO;
  - при удержании `TXE_N = 0`;
  - с выдачей одного слова за такт `CLK` без возврата в idle между соседними словами burst-последовательности.

Требуемое поведение приема из FT601:
- чтение допускается только при `RXF_N = 0` и наличии места во внутреннем RX FIFO;
- принятые `DATA` и `BE` должны фиксироваться в домене `CLK`;
- после приема слова должен формироваться запрос записи в RX FIFO.
- прием выполняется burst-режимом:
  - `OE_N` и `RD_N` удерживаются активными во время непрерывного чтения;
  - по одному слову на такт `CLK`, пока `RXF_N = 0` и прием не заблокирован внутренним `full_fifo`.

Важно для дальнейшей доработки:
- верификация `fifo_fsm.v` должна отдельно подтверждать соответствие временным диаграммам из `DS_FT600Q-FT601Q-IC-Datasheet.pdf`;
- при sign-off необходимо отдельно проверить отсутствие смещения данных на TX-path относительно команды чтения из TX FIFO;
- текущая реализация burst существенно проще reference design FTDI и не содержит полноценного prefetch FIFO, как в `mst_pre_fet.v`.

### 8. Модуль `fifo_rx_ctrl.v`

Назначение:
- чтение принятых RX-слов из внутреннего RX FIFO;
- фиксация sticky-ошибок RX/TX со стороны FT601-domain;
- декодирование служебных команд, приходящих с ПК.

Поддерживаемые команды:
- `32'h00000001` при `BE = 4'hF` — очистка ошибки TX;
- `32'h00000002` при `BE = 4'hF` — очистка ошибки RX;
- `32'h00000003` при `BE = 4'hF` — очистка обеих ошибок.

Требуемое поведение:
- RX FIFO читается автоматически, пока он не пуст;
- очистка ошибки TX передается в write-domain через toggle-сигнал и двухтактную синхронизацию;
- ошибки RX фиксируются sticky-флагом.

## Ошибки и их обработка

В проекте используются sticky-ошибки, которые сохраняются до явной очистки.

### Ошибки TX

Фиксируются при событиях:
- попытка записи в полный TX FIFO;
- попытка чтения TX FIFO из пустого состояния.

После фиксации ошибки TX:
- блокируется дальнейшая запись слов из `packer8to32` в TX FIFO;
- очистка выполняется только внутренней RX-командой от хоста.

### Ошибки RX

Фиксируются при событиях:
- переполнение RX FIFO;
- underflow RX FIFO, если этот контроль включен.

После фиксации ошибки RX:
- признак ошибки сохраняется sticky-флагом;
- очистка выполняется только внутренней RX-командой от хоста.

## Требования к тестбенчам и верификации

### Общий принцип

Проверка должна выполняться по этой спецификации.
Тестбенч не должен подстраиваться под случайные особенности реализации, если они не зафиксированы в этом документе как часть требований.

### Что обязательно должно проверяться

1. Входной тракт `LVDS -> packer`
- регистрация входных данных по `posedge` входного такта;
- игнорирование байтов при `strob = 0`;
- корректная упаковка 8 бит -> 32 бита;
- корректный порядок байтов внутри 32-битного слова.

2. Асинхронный TX FIFO
- корректная передача слов между двумя тактовыми доменами;
- корректная работа `full` и `empty`;
- корректная фиксация `overflow` и `underflow`.

3. TX-передача в FT601
- отсутствие передачи при `TXE_N = 1`;
- корректная выдача `WR_N`, `DATA`, `BE`;
- корректное переключение направления `DATA/BE` через `drive_tx`;
- корректное чтение слов из TX FIFO по запросам `fifo_pop`;
- корректная непрерывная burst-передача без одиночных пауз между соседними словами при `TXE_N = 0`;
- соответствие временным диаграммам FT601.

4. RX-прием из FT601
- отсутствие чтения при `RXF_N = 1`;
- отсутствие чтения при запрете со стороны внутреннего `full_fifo`;
- корректная фиксация `DATA` и `BE`;
- корректная запись принятых слов во внутренний RX FIFO;
- корректная последовательность `OE_N -> RD_N -> capture -> fifo_append`;
- корректное burst-чтение без возврата в idle между соседними словами при `RXF_N = 0`.

5. Обработка ошибок
- фиксация sticky-ошибок;
- блокировка TX-потока после ошибки;
- корректная очистка ошибок командами `0x00000001`, `0x00000002`, `0x00000003`.

### Требования к текущим тестбенчам

#### `fpga/lvds_tb.v`

Назначение текущего тестбенча:
- модульно-интеграционная проверка тракта
  - `LVDS.v`
  - `packer8to32.v`
  - `fifo_dualport.v`
  - `sram_dp.v`
  - `fifo_fsm.v`

Особенности `lvds_tb.v`:
- тестбенч использует упрощенную модель окружения FT601;
- проверка `fifo_fsm.v` выполняется без полного top-level, но с реальными сигналами `TXE_N`, `RXF_N`, `WR_N`, `RD_N`, `OE_N`, `DATA`, `BE`;
- tri-state поведение внешней шины не проверяется в `lvds_tb.v`, оно относится к уровню `fifo_top.v`.

Обязательные сценарии для `lvds_tb.v`:
- `test_fifo_underflow`:
  - проверка фиксации `underflow` при чтении пустого FIFO;
- `test_fifo_overflow`:
  - проверка фиксации `overflow` при записи в полный FIFO;
- `test_nominal_direct_fifo_path`:
  - проверка номинального тракта `LVDS -> packer -> FIFO -> SRAM` без участия `fifo_fsm`;
- `test_fsm_tx_path`:
  - проверка, что `fifo_fsm` не передает данные при `TXE_N = 1`;
  - проверка передачи слов из TX FIFO при `TXE_N = 0`;
  - проверка `WR_N`, `DATA`, `BE`, `drive_tx` на TX-пути;
  - проверка burst-передачи без возврата в idle между соседними словами;
- `test_fsm_rx_blocked_by_full`:
  - проверка отсутствия чтения FT601, если внутренний приемный FIFO помечен полным;
- `test_fsm_rx_path`:
  - проверка чтения слов от FT601 по `RXF_N`;
  - проверка захвата `DATA` и `BE`;
  - проверка выдачи `fifo_append` после принятого слова;
  - проверка burst-чтения при последовательных RX-словах.

Текущее правило моделирования входной паузы в `lvds_tb.v`:
- последовательность `FF000000FF000000FF000000FF000000` используется только как тестовый шаблон;
- сама RTL-логика не распознает этот шаблон;
- невалидность входных данных для DUT задается исключительно через `strob = 0`.

#### `fpga/lvds_tb_top.v`

Требуемое назначение:
- полная проверка `fifo_top.v`;
- моделирование handshake с FT601 в обе стороны;
- проверка RX-команд и sticky-ошибок;
- проверка tri-state поведения шины `DATA/BE`;
- проверка внутренних блоков `fifo_rx_ctrl.v` и `fifo_tx_ctrl.v` через интерфейс верхнего уровня.

## Известные ограничения текущей версии

На момент обновления документа действуют следующие ограничения:
- имя модуля `LVDS.v` историческое; фактически на вход Spartan-6 приходят уже однополярные сигналы `LVCMOS33`;
- передача неполного 32-битного слова в FT601 не поддерживается;
- `BE` на TX-пути формируется только как `4'hF`;
- реализован только одноканальный режим FT601;
- текущий `fifo_fsm.v` проще reference `master_fifo` от FTDI:
  - нет multi-channel логики;
  - нет out-of-band/sideband режима;
  - нет отдельного prefetch FIFO уровня `mst_pre_fet.v`;
  - нет логики `remain`/повторной отправки, как в полном reference design;
- максимальная производительность FT601-интерфейса пока не рассматривается как закрытый критерий приемки; сначала требуется функциональная корректность и корректная верификация протокола.

## Состав ключевых файлов проекта

- `fpga/fifo_top.v` — финальный верхний модуль проекта
- `fpga/LVDS.v` — входная буферизация и регистрация входного параллельного потока
- `fpga/packer8to32.v` — упаковка 8-битных байтов в 32-битные слова
- `fpga/fifo_dualport.v` — асинхронный FIFO на Gray-указателях
- `fpga/sram_dp.v` — двухпортовая синхронная память для FIFO
- `fpga/fifo_fsm.v` — FSM интерфейса FT601
- `fpga/fifo_tx_ctrl.v` — контроль TX-ошибок и разрешения записи в TX FIFO
- `fpga/fifo_rx_ctrl.v` — прием RX-команд и контроль RX/TX sticky-ошибок
- `fpga/lvds_tb.v` — модульно-интеграционный тестбенч входного тракта и `fifo_fsm.v`
- `fpga/lvds_tb_top.v` — верхнеуровневый тестбенч `fifo_top.v`

## Рекомендуемый порядок sign-off

1. Подтвердить корректность `LVDS -> packer -> TX FIFO` и базового поведения `fifo_fsm` на `fpga/lvds_tb.v`.
2. Подтвердить корректность FT601 TX/RX handshake и логики верхнего уровня на `fpga/lvds_tb_top.v`.
3. Подтвердить обработку sticky-ошибок и их очистку командами от хоста.
4. После этого переходить к синтезу, constraints и проверке на железе.
